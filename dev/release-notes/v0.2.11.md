# AI Public Squares v0.2.11 — Pre-release (Peer Lifecycle Fix Build)

A pre-release patch on top of 0.2.10, cut from real cross-machine UAT (Mac + Windows). Not a release pass. It hardens the Project Peers confirmation lifecycle and reframes the setup "role" question; it adds no new features.

## Status

- npm latest pre-release target: `@adamchanadam/aps@0.2.11`.
- Same Reliable Peer Handoff scope as 0.2.9 / 0.2.10 (Project Peers + Sent Status).

## Fixed

- **Peer confirmation now tracks real participation.** Publishing or consuming a packet from your own machine self-confirms your own peer card. Previously a peer became `confirmed` only by running `aps init`, so an agent who had already published / consumed could stay `provisional` and block the other side from replying. (Found in real Mac UAT: an active sender could not be replied to.) Self-confirm only ever updates the executing agent's own card — never the counterpart's — and is skipped when identity is overridden via `--from` / `--agent-id`.
- **`publish --to` recipient gate is now three-way** instead of confirmed-only: `confirmed` → send; `provisional` but with real activity evidence (own lane has a publish event, or own ack has a consumed entry) → send with a status-pending warning; `inactive` / unregistered / no activity → block. A bare lane / ack no longer counts as "joined", because `init` and `peer add` create those skeletons for both sides.
- **Setup "role A/B" reframed.** The interactive `aps init` question no longer calls it "your role"; it asks which setup direction you are (first to set up / build the Hub, or joining via a starter pack) and states plainly that it does not limit who can send or receive. `aps config` now shows "設定起手方向" instead of "🎭 角色", with a note that send/receive is symmetric. After you enter the Hub path, if the project already exists and the named counterpart has already set up, the confirm screen shows a gentle hint that you are most likely the joiner — without reordering the setup questions.

## Authorization boundary (reaffirmed)

Who may send / receive is decided only by `agent_id`, packet `to`, ack, and activity evidence — never by `role`. `role` remains an internal setup-direction hint for starter-pack / bridge templating; it has no permission meaning. Roadmap §4.2 has been updated to match.
