1.開始之前 — 你會走過怎樣的一條路

由「兩位用戶之電腦皆未接通」開始,行至「第一次跨機交接跑通,雙方可試行日常收發」為止。總時間約 60-90 分鐘,假設 Google Drive 已 mount 妥當。

想由 AI 帶你一步一步做:在已安裝 APS skill 的 AI 工具中直接說 set up APS 或「教我用 APS」。本頁是完整深入參考,適合想逐句核對設置、跨機驗證與補救邏輯的人閱讀;不是唯一入口。
Adam / Jay 只是兩個 example 用戶名。本 repo 為通用 APS template,不綁定任何特定項目或人物。下文以「Adam」、「Jay」做敘事人物講故事,實際落地時:Adam 對應 發起接駁的一方(User A),Jay 對應 對方(User B)。雙方自行決定 agent_id、項目代號、工作目錄路徑。試演留下之 demo workspace 與 demo slug 僅為 MVP 驗證所用之 fixture,並非 default value。

五個方格之詳細做法,下面段落逐個展開。實線表示須等上一步完成;虛線表示可推遲。

本教學寫予真人閱讀,結合敘事與實際範例。執行步驟之精準版本(包括 AI 程式須依照之技術細節)為 docs\plans\2026-05-21-aps-phase4-plan.md(AI / 維護者層 .md 檔,瀏覽器開為 plain text;由 AI 程式直接讀)。
用語慣例:方括號 placeholder 須 substitute。教學中所有方括號之短語 — 例如 [項目代號]、[題目]、[時間戳]、[工作目錄]、[Jay 部機之共享資料夾絕對路徑] — 皆為 placeholder,屬 forcing function。複製至規則檔、發送 WhatsApp 之前,須逐個換為你們第 5 節拍板之實際值。協定本身通用,教學文字不列具體 value;具體 value 由雙方自行決定。
Placeholder 樣本 — 形狀與範例。每個 placeholder 之 substitute value 應為何種樣式,下列以幾個 hypothetical 範例做 reference(下列範例皆非實際 demo 之 slug,純屬樣式示例):
  • [項目代號] — snake_case 短名(英文小寫加底線加數字,長度 30 字內)。例:branding_2026、logo_refresh_q2、internal_pitch、cobeing_website。協定上鎖小寫;Windows NTFS 雖不分大小寫,但 Google Drive 同步至其他系統有機會出 bug,故穩定小寫。不要放中文、空格或特殊符號。
  • [Adam 部機之共享資料夾絕對路徑] / [Jay 部機之共享資料夾絕對路徑] / [Hub 絕對路徑] — 由磁碟字母開始之 absolute path,指向你部機掛載 Hub 之實際位置;最後一段必然為 AI_Public_Squares。例:G:\Cloud\AI_Public_Squares、D:\Work\AI_Public_Squares、H:\我的雲端硬碟\Marketing\AI_Public_Squares。雙方各自寫自己部機之版本,磁碟字母與中間段可不同。
  • [工作目錄] — 你部機之項目工作目錄絕對路徑(裝有 Agent Handoff Kit 之 root,即開啟新對話予 AI 之資料夾)。例:C:\Work\branding、D:\Projects\client_x。
  • [題目] — snake_case 短主題,描述此包做什麼。例:logo_convergence、brand_voice_v2、color_semantics、tagline_review。同樣鎖小寫加底線。
  • [時間戳] — UTC ISO 8601 compact 格式 YYYYMMDDTHHMMSSZ,反映你出包之時刻之 UTC 時間。例:20260520T103000Z(即 2026 年 5 月 20 日 10:30:00 UTC)。AI 程式自動生成,無需手寫。

2.安裝先決事項 — 3 件須先做

完成此 3 件事之前,後面之步驟全部無法執行。雙方各自於自己部機完成。

第一件 — 安裝 Agent Handoff Kit(治理基礎)

APS 之檔案結構依靠 Agent Handoff Kit 之 managed-core block、doctor 與 upgrade 機制。未先安裝,APS 之檔案結構無法獨立運作。

無 Agent Handoff Kit = APS 無法運作。此為硬性前置。

前往 https://github.com/Adamchanadam/agent-handoff-kit,依其指引安裝。本工具透過 npx 機制全域可用,毋須額外 install — 於 §3 將會於你之工作目錄中執行其 init 命令。

第二件 — 安裝 Google Drive 桌面版

  • 前往 https://www.google.com/drive/download/ 下載桌面版
  • 登入後將共享之資料夾右鍵設為「離線存取」(此設定保證 AI 程式無需經網絡即可讀寫 Hub)

第三件 — 安裝 Claude Code

  • 前往 https://claude.com/code 下載
  • 登入後,於命令行可執行 claude 命令啟動對話 session
前期測試邊界。npm latest 0.2.2 pre-release 之主路徑是先於項目工作目錄執行 npm install --save-dev @adamchanadam/aps,再用 npx aps init --dry-run 預覽設置,確認後以 npx aps init --hub-root ... 建立技能、初始 Hub skeleton、Bridge Pack、starter pack 與 .aps/config.json 本地設定。設置後日常命令可省略 Hub / project / agent 長參數。CLI 已有最小發佈、收件、消化、回覆、收結往返測試,並完成一次 Adam ↔ Jay 真實 Google Drive 跨機往返驗證。自然語言日常操作與補救流程仍未完整產品化。
3 件事尚未完成?各個官方網頁皆有逐步指引。先完成全部 3 件,再返回繼續第 3 節。

3.準備你的項目工作目錄

§2 3 件先決事完成之後,仍須 set up 你的項目工作目錄。本節做兩件事:創建一個 NEW folder 作為此合作項目之 root,並於該 folder 中初始化 Agent Handoff Kit。後面 §6、§7 全部步驟皆於此工作目錄之中進行。

工作目錄為此合作項目之專用 folder,要 create 一個新 folder,不要 reuse 既有 project folder。例:你做 brand book,工作目錄可命名為 branding_2026;你做 design,可命名為 logo_refresh。Bridge Pack fixture 將由 §6 步驟 2 之 npx 命令寫入此工作目錄之 dev/rules/。

3.1 創建工作目錄

於命令行執行(將 branding_2026 換為你項目之實際名稱;Windows 範例):

mkdir C:\Work\branding_2026
cd C:\Work\branding_2026

macOS / Linux 範例:mkdir -p ~/Work/branding_2026 && cd ~/Work/branding_2026。
路徑可揀任何位置,但建議避免:中文字符路徑(避免 Drive 同步邊緣 bug)、含空格之路徑(避免 shell escape 麻煩)、需要 admin 權限之系統位置(例 C:\Program Files\)。

3.2 於工作目錄中初始化 Agent Handoff Kit

於工作目錄之命令行執行:

npx --yes @adamchanadam/agent-handoff-kit@latest init

命令會先列出寫入 / 合併計劃。檢查目標路徑正是本節建立之工作目錄後,輸入 yes 確認寫入。完成後該 folder 會多出一系列 governance 檔案,包括以下關鍵入口:

  • AGENTS.md
  • dev/SESSION_HANDOFF.md
  • dev/SESSION_LOG.md
  • dev/PROJECT_INDEX.md
  • dev/RULE_PACKS.md
  • dev/DOC_SYNC_REGISTRY.md

此為 APS 後續步驟所依賴之治理基礎結構。

3.3 跑 doctor 驗證

於工作目錄之命令行執行:

npx --yes @adamchanadam/agent-handoff-kit@latest doctor

預期 output:Status: passed。不同 Agent Handoff Kit 版本之檢查項數會改變,以 status 為準;若 status 為 failed,將完整 output 貼予 Agent Handoff Kit 之 GitHub issue 求助。

Agent Handoff Kit 已於此工作目錄寫入治理基礎;關鍵入口檔案存在;doctor status 為 passed;工作目錄為此合作項目專用之 NEW folder。完成此節進入 §4。

4.概念熱身 — 教學中常見之詞

後續各節會出現以下詞;此節一次過解釋,後面遇到時不再重複定義。

詞意思
Hub(共享資料夾) Google Drive 上雙方都見到之資料夾,固定名為 AI_Public_Squares。雙方 AI 程式皆視為本地資料夾直接讀寫,Google Drive 處理同步。
Bridge Pack(橋接小規則) 一份 markdown 規則檔(dev/rules/aps-bridge.md),教 AI 程式如何解讀 Hub 上之檔案。每位用戶之工作目錄都需要一份。此教學第 6 / 7 節即係安裝此檔。
packet(交接包) 一個資料夾,內含一份 packet.md(交接之主內容)+ 可選 attachments/。一旦發出即不再修改;若需更新則出 v2 並標明取代哪一版。
outbox(出包紀錄) 每位用戶在自己 lane 之 outbox.log.md,逐行紀錄每次出包、改版、收結、撤回。append-only。
ack(消化紀錄) 每位用戶之 _ack/<agent_id>.ack.json,記下自己消化過對方哪些 packet。
lane(通道) Hub 上之子資料夾 from_<agent_id>/。每位用戶各自寫自己 lane,不寫對方 lane。Drive 同步衝突由此結構性避免。
Agent Adam / Agent Jay 運行於 Adam / Jay 部機之 AI 程式(本教學中用 Claude Code)。與 Adam / Jay 用戶本人之動作分開記錄。
doctor(健康檢查) Agent Handoff Kit 之健康檢查命令(npx --yes @adamchanadam/agent-handoff-kit@latest doctor),驗證 governance 檔案結構正確。每次設置完應跑一次確認綠燈。
kit Agent Handoff Kit 之簡稱。

5.先傾妥幾件事

不動任何檔案。十五分鐘左右之電話或 WhatsApp 傾妥即可 — 你們實際要拍板者,只有三件;其餘四件皆有合理默認,沿用即可。此步為用戶之間之事,不涉及兩個 AI 程式。

須拍板之三件事

須傾之事建議備註
項目代號 選新之 snake_case 短名(例:branding_2026、logo_refresh) 選全新 slug 最為乾淨 — 不需清任何已有 lane 資料,協定亦不受影響。若想沿用某個共享資料夾中已有之 slug(例如曾做過試演或過往項目使用過),須先清舊資料(刪舊 packets、reset outbox)— 多了一個 step,通常不划算。樣本之完整形狀規則見第 1 節「Placeholder 樣本」。
Jay 部機路徑 Jay 自選工作目錄並於自己部機 mount Hub 例:工作目錄 D:\Work\[項目代號]_Jay\(須完成 kit 之初始化,doctor 通過);Hub 之磁碟字母可能不同(例 H:\ 而非 G:\),此為合理 — 小規則檔內各自寫實際路徑,協定本身不受影響。
大檔試跑是否進行 建議進行,可推遲 第一份真實之逾 50 MB 資產出現之前必須進行一次。若近期皆為文字材料,可遲一步補做。

預設默認之四件事(不需改即可)

項目默認值何時須改默認
Adam agent 名 自選(snake_case,1-30 字內;MVP fixture 使用 adam 做範例) 選短、stable、容易讀即可;無需跟 fixture。
Jay agent 名 自選(snake_case;MVP fixture 使用 jay 做範例) 同上。
共享資料夾中已有之 lane 資料 留底不動(選新 slug 之前提下,已有 lane 自然不受影響) 只有選沿用某個已有 slug 之時才有得選清 / 留;選新 slug 即不關事。若共享資料夾剛啟動、未有任何 lane,此項 N/A。
Adam 部機 Hub 路徑與 Drive 分享權限 Adam 那邊部機掛載 Hub 之實際絕對路徑([Adam 部機之共享資料夾絕對路徑])+ Drive 上層 AI_Public_Squares 資料夾 share 予 Jay 有寫權 若 Drive sharing 設於 slug 子資料夾而非 AI_Public_Squares root,新 slug 須新增 share。若 Hub 尚未掛載或 Drive 尚未 share,須先補完此兩件機器 setup 方能繼續。
雙方傾妥之後

Adam 打開技術版計劃文檔最末之參數表,將拍板之事與確認之默認填齊。此刻即此節完成。

6.Adam 接駁自己部機

此節 Adam 於自己部機,要求 Agent Adam 協助執行;Jay 無需在場。npm latest 0.2.2 pre-release 主路徑先用 npx aps init --hub-root ... 建立技能、Hub skeleton、Bridge Pack 與 .aps/config.json,再補登記治理檔。完成後 Adam 之工作目錄已接好橋接小規則,但尚未交收任何 packet。

主路徑:CLI 先行,手動備用。npm latest 0.2.2 pre-release 已提供 aps init 建立技能、Hub skeleton、Bridge Pack、starter pack、本地設定與短命令日用流程。下面第 2 步以 CLI 為主;第 3 至第 6 步仍用 Agent Adam 補登記治理檔,因為目前 CLI 尚未自動改寫你的 project governance。舊 tools/aps-onboard.ps1 只作維護者備用路徑,新用戶毋須先學。
Adam與Agent Adam逐步動作

  1. Agent Adam 做唯讀檢查

    此步之目的:正式驗證 §2 之 3 件先決事完整、§3 之工作目錄已 init、kit 健康、Claude Code 可與 Agent Adam 對話。Agent Adam 不會改任何檔。

    1.1 打開命令行

    Windows:打開 PowerShell 或 Windows Terminal。macOS / Linux:打開 Terminal。

    1.2 cd 入你之工作目錄

    於命令行執行(將路徑換為你 §3 中創建之實際路徑;Windows 範例):

    cd C:\Work\branding_2026

    macOS / Linux 範例:cd ~/Work/branding_2026。如尚未創建工作目錄,先返回 §3「準備你的項目工作目錄」完成創建。

    1.3 啟動 Claude Code 對話

    於工作目錄中執行:

    claude

    此命令啟動 Claude Code 對話 session。後續所有與 Agent Adam 之互動皆於此對話中進行。如命令行報「未找到 claude 命令」,§2 第三件之 Claude Code 安裝未完成,先返回 §2 安裝。

    1.4 將以下 prompt 貼予 Agent Adam

    於 Claude Code 對話中複製貼上以下 prompt(你不需要明白其中之技術細節,Agent Adam 會處理):

    請做以下事但不要改任何檔:

(a) 列出工作目錄之 root 內是否存在以下 5 個關鍵入口檔案:
    AGENTS.md、dev/SESSION_HANDOFF.md、dev/RULE_PACKS.md、
    dev/PROJECT_INDEX.md、dev/DOC_SYNC_REGISTRY.md

(b) 執行 `npx --yes @adamchanadam/agent-handoff-kit@latest doctor`,
    將 output 完整報予我

(c) 執行 `git status`,將 output 報予我;若 git 未初始化,
    告知我並建議跑 `git init`

報告 (a) (b) (c) 之結果,然後停手等我指示。

    1.5 Agent Adam 報出之內容應對照以下表

    檢查項預期結果若不符合,如何處理
    (a) 5 個關鍵入口檔案 5 個 file 皆存在 §3 之 init 命令未跑齊,或尚未於互動確認中輸入 yes。返回 §3.2 補做。
    (b) doctor output Status: passed(檢查項數依 kit 版本而定) 見下方「若 doctor 失敗」
    (c) git status 顯示分支與工作樹狀態(已 init);或 Agent Adam 建議 git init(未 init) 未 init 可選,但建議跑 git init,APS 之檔案改動方可追溯
    若 doctor 失敗:常見原因為 kit 版本過舊或 governance 檔案不完整。Doctor output 會顯示版本與失敗項。若版本落後,將以下 prompt 貼予 Agent Adam: 
    請執行 npx --yes @adamchanadam/agent-handoff-kit@latest upgrade,將完整 output 報予我。
    Upgrade 完成後重做 1.4 步驟,doctor status 須為 passed 方可繼續第 2 步。其他失敗原因,將 doctor output 連同 Agent Adam 之報告貼予 Agent Handoff Kit 之 GitHub issue 求助。
    5 個關鍵入口檔案皆存在;doctor status 為 passed;git 已 init(或已決定不 init);Agent Adam 無做任何檔案改動。完成此步即可進入下一步(複製 Bridge Pack)。

  2. 用 CLI 建立 APS skill、Hub skeleton 與 Bridge Pack

    此步之目的:用 npm latest 0.2.2 pre-release CLI 一次建立你部機需要的 APS skill、共享 Hub skeleton、本地 dev/rules/aps-bridge.md、.aps/config.json,以及給 Jay 的 starter pack。設置後可使用短命令。Bridge Pack 為一份 markdown 規則檔,教 Agent Adam 如何讀寫 Hub。

    前期測試邊界:npm latest 0.2.2 pre-release 主路徑為先執行 npm install --save-dev @adamchanadam/aps,再用 npx aps init --dry-run 預覽寫入位置,確認後才移除 --dry-run 真正設置。npx aps bridge-pack 仍可作維護者取得 fixture 之備用命令,但新用戶應優先跟隨本節之 init 流程。此步會額外保存本地設定,方便後續短命令日用。

    2.1 將以下 prompt 貼予 Agent Adam

    將 [項目代號]、[Adam 部機之共享資料夾絕對路徑]、[Adam agent id]、[Jay agent id] 換為 §5 拍板之實際值之後,複製貼上。若沿用 narrative 範例,Adam agent id 為 adam,Jay agent id 為 jay。

    請執行以下事:

(a) 先執行:
      npm install --save-dev @adamchanadam/aps

(b) 先 dry-run,不寫檔,只讓我確認計劃:

驗證「逾 50 MB 之檔不放入包內,改放外面再於包中說明位置」此條規矩。試演未試過,第一份真實大檔出現之前必須試一次。

Adam與Agent Adam動作
  1. Adam 自行準備一份逾 50 MB 之 dummy 檔。可為 padded zip、真實大 PSD,或 PowerShell 執行 fsutil file createnew test.bin 62914560 製造一個 60 MB 之空檔。
  2. Adam 自行將該檔放入共享資料夾之 sibling 位:[共享資料夾]\_external\[題目]\[檔名](若 _external\ 不存在則建立)。
  3. Adam 要求 Agent Adam 出一個 packet,packet 中之 YAML ssot_refs: 寫回此絕對路徑:
ssot_refs:
  - "[Hub 絕對路徑]\\_external\\[題目]\\big_asset.psd"

複製出來後,將 [Hub 絕對路徑] 換為你部機掛載 Hub 之實際 absolute path(例 G:\Cloud\AI_Public_Squares),[題目] 換為此 packet 之 snake_case 短主題(例 logo_convergence);\\ 雙反斜線為 YAML escape,不要改為 single backslash。

  1. Packet 內正文寫清楚:檔為何、為何放外面、Jay 應如何打開。Adam 撰寫,Agent Adam 入檔。
  2. Agent Adam 收尾報告中標明:「Externalized: big_asset.psd (XX MB) at [Hub 絕對路徑]\_external\…」。
  3. Adam 自行以 WhatsApp 通知 Jay,與正常 packet 同。
Jay與Agent Jay收方
  1. Jay 正常開啟新對話、貼模板。Agent Jay 啟動流程列出 packet,同時要報出外置檔之路徑予 Jay 閱讀。
  2. Jay 親手於自己部機 Drive 中以 Explorer / 設計工具打開外置檔,確認檔可讀。Agent Jay 不負責此步,因其只讀寫 Hub 之內檔案。
Packet 內 attachments\ 無任何檔逾 50 MB;外置檔之通知出現於 Agent Adam 收尾報告與 Agent Jay 啟動報告;Jay 可以 Drive mount 打開外置檔。

10.日常試行使用

§6 至 §8 跑通後,APS pre-release 試行接駁已完成。日常試行用法為一個重複拍子,每次往返皆為此十步。每個 pill 標明為用戶動作或 agent 動作。

十步走完即一次交接收結。實邊框為人,虛邊框為程式。一日不一定走齊十步,可中途 pause(例:Jay 慢慢做)。

多條交接同時進行則為兩條此十步並行,共享資料夾之事件記錄與消化記錄表會記齊。

日常 WhatsApp 短訊版。日常運作中,WhatsApp 之短訊無需再貼完整模板予對方再貼予 AI 程式。一句「Hub 有新嘢」已足夠通知對方;對方於自己工作目錄之現有對話中講一句「check Hub」即可,AI 程式立即撿包(此為默認之日常流程)。開啟新對話並貼整段模板為備用做法,僅限於第一次跨機(§8)或想重新開始。完整英文模板(§8 之 APS Hub has new traffic...)保留予首次跨機交接做啟蒙教學;日常跑通之後一句「check Hub」即可。
頻率估計:Adam → Jay 方向通常一日一兩次。Jay 回包可能即日內,可能等次日。一條交接之 wall-clock 約 12-48 小時。緊急之交接,WhatsApp 加一句「即刻」即可,協定本身不阻你即時催。

11.中途想交換,不等對話結束

§8 與 §10 中所講「對話收尾時要求 agent 出包」,此為順手做之時機,並非硬性。Adam 或 Jay 之對話中途任何時刻,只要內容 ready,皆可即時交換。

Adam與Agent Adam· 想中途即時出包

Adam 於對話中途講「我想現在 send 一份予 Jay」,要求 Agent Adam 依出包流程做。Agent Adam 立即寫 packet 入共享資料夾 + 加事件記錄。Packet 為不可變之 snapshot,Adam 之後於對話中之工作,不會倒灌入已出之 packet。

Adam 親自立即發 WhatsApp 模板予 Jay,然後繼續下面之工作。

Jay與Agent Jay· 想中途撿剛到之 packet

Jay 於對話中途收到 Adam 之 WhatsApp,無需 close 對話再開啟新對話。Jay 對 Agent Jay 講一句官方觸發詞:

  • check Hub — 官方觸發詞,最短最直接。Agent 認 case-insensitive(check hub / Check Hub 皆可)。

另兩句同義講法 — 睇下 Hub 有冇新嘢、未消化 — Bridge Pack 皆認,作寬鬆補救(句中含 Hub 連同 新 / 未消化 / check 任一字眼,即觸發同一個處理流程)。慣用哪句皆可,但官方一句為 check Hub。

觸發之後 Agent Jay 立即去共享資料夾撿,計算差集,將新待辦呈現予 Jay。Jay 決定做與不做、何時做。

此為手動觸發,並非自動推送。Agent Jay 不主動去 poll 共享資料夾 — 它只於 Jay 開啟新對話啟動時、或 Jay 中途要求一句時方去 check。換言之,Jay 不開對話 / 不要求,Agent Jay 永遠不知有新內容。
真的想自動推送(部機自動彈通知)?此事需要本地 daemon 監察共享資料夾中 outbox 之修改時間,超出純檔案範圍之設計,加重 coupling dependency。設計取捨上,WhatsApp 之人手叫一聲為刻意保留,讓收方控住何時被打斷;自動推送之 cost 為噪音與中斷工作流。值不值得視乎雙方實際使用之頻率與緊急程度,Phase 4 落地之後再評估。

12.出錯處理

12.1 出現 Drive 衝突副本(file (1).md 之類)

理論上不應發生,因有單寫道規矩。Bridge Pack 於每次啟動與 check Hub 觸發時皆自動掃描共享資料夾,掃到任何含 conflict 字眼之檔即時報予你閱讀 — 用戶無需主動跑命令查。若 Agent 報過,或你自行肉眼看到:

  • Adam 或 Jay 不要即時手動刪。
  • 停手,記下衝突檔之完整路徑與名稱。
  • 對照單寫道規矩,找出哪個 agent 程式寫了對方通道之檔(通常此即為根源 — Agent Adam 寫了 from_jay,或 Agent Jay 寫了 from_adam)。
  • 修正根源之後,先決定保留哪個版本(通常為原寫者之版本),衝突副本移走或更名為 .investigated。

12.2 出包但內容錯 / 想撤回

用戶發現錯,用戶與 agent 一同 check 對方是否登記,然後用戶決定,agent 執行。

撤回之行寫法(由 agent 寫):

[時間] | withdraw | [時間戳]__[題目] v1 | reason:作者要求撤回原版

12.3 Agent 撈錯通道(寫了對方通道、寫了對方消化記錄表)

邊界硬規矩應該攔到。若真的漏了:

  • 用戶立即停手,不要繼續要求 agent 出包。
  • 用戶檢查該誤寫究竟為 placeholder(可清)還是真實內容(由用戶決定如何處理)。
  • 用戶決定是否手動清除誤寫;事件 + 原因記入 SESSION_LOG.md(由 agent 寫),讓下次開啟新對話可追溯。

12.4 忘記收結,thread 永遠掛緊待辦

無大事。下次任何一邊開啟新對話,agent 啟動流程列其為待辦,用戶要求 agent append 一行 close 事件即可。close 完之後再開啟新對話即為無待辦狀態。

12.5 Drive 同步延遲逾五分鐘

幾乎一定不是協定問題。Adam 或 Jay 用戶依序檢查:

  1. Drive 桌面版是否正在運行、是否已 sign-in?
  2. 共享資料夾是否已設「離線存取」?若僅為「串流」可能有延遲。
  3. 網絡頻寬是否足夠?大檔則可能須再等。
  4. 對方部機是否 online?Drive 不會主動 push,須 receiver 那邊之 Drive 客戶端 sync 方能見到。
