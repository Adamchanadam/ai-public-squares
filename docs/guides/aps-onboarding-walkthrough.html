1.開始之前 — 你會走過怎樣的一條路

由「兩位用戶之電腦皆未接通」開始,行至「第一次跨機交接跑通,雙方可日常使用」為止。總時間約 60-90 分鐘,假設 Google Drive 已 mount 妥當。

Adam / Jay 只是兩個 example 用戶名。本 repo 為通用 APS template,不綁定任何特定項目或人物。下文以「Adam」、「Jay」做敘事人物講故事,實際落地時:Adam 對應 發起接駁的一方(User A),Jay 對應 對方(User B)。雙方自行決定 agent_id、項目代號、工作目錄路徑。試演留下之 demo workspace 與 demo slug 僅為 MVP 驗證所用之 fixture,並非 default value。

五個方格之詳細做法,下面段落逐個展開。實線表示須等上一步完成;虛線表示可推遲。

本教學寫予真人閱讀,結合敘事與實際範例。執行步驟之精準版本(包括 AI 程式須依照之技術細節)見 技術版計劃文檔,兩份對讀則最為完整。
用語慣例:方括號 placeholder 須 substitute。教學中所有方括號之短語 — 例如 [項目代號]、[題目]、[時間戳]、[工作目錄]、[Jay 部機之共享資料夾絕對路徑] — 皆為 placeholder,屬 forcing function。複製至規則檔、發送 WhatsApp 之前,須逐個換為你哋第 4 節拍板之實際值。協定本身通用,教學文字不列具體 value;具體 value 由雙方自行決定。
Placeholder 樣本 — 形狀與範例。每個 placeholder 之 substitute value 應為何種樣式,下列以幾個 hypothetical 範例做 reference(下列範例皆非實際 demo 之 slug,純屬樣式示例):
  • [項目代號] — snake_case 短名(英文小寫加底線加數字,長度 30 字內)。例:branding_2026、logo_refresh_q2、internal_pitch、cobeing_website。協定上鎖小寫;Windows NTFS 雖不分大小寫,但 Google Drive 同步至其他系統有機會出 bug,故穩定小寫。不要放中文、空格或特殊符號。
  • [Adam 部機之共享資料夾絕對路徑] / [Jay 部機之共享資料夾絕對路徑] / [Hub 絕對路徑] — 由磁碟字母開始之 absolute path,指向你部機掛載 Hub 之實際位置;最後一段必然為 AI_Public_Squares。例:G:\Cloud\AI_Public_Squares、D:\Work\AI_Public_Squares、H:\我的雲端硬碟\Marketing\AI_Public_Squares。雙方各自寫自己部機之版本,磁碟字母與中間段可不同。
  • [工作目錄] — 你部機之項目工作目錄絕對路徑(裝有 Agent Handoff Kit 之 root,即開啟新對話予 AI 之資料夾)。例:C:\Work\branding、D:\Projects\client_x。
  • [題目] — snake_case 短主題,描述此包做什麼。例:logo_convergence、brand_voice_v2、color_semantics、tagline_review。同樣鎖小寫加底線。
  • [時間戳] — UTC ISO 8601 compact 格式 YYYYMMDDTHHMMSSZ,反映你出包之時刻之 UTC 時間。例:20260520T103000Z(即 2026 年 5 月 20 日 10:30:00 UTC)。AI 程式自動生成,無需手寫。

2.安裝前置事項 — 4 件須先做

完成此 4 件事之前,後面之步驟全部無法執行。雙方各自於自己部機完成。

第一件 — 安裝 Agent Handoff Kit(治理基礎)

APS 之檔案結構依靠 Agent Handoff Kit 之 managed-core block、doctor 與 upgrade 機制。未先安裝,APS 之檔案結構無法獨立運作。

無 Agent Handoff Kit = APS 無法運作。此為硬性前置。

前往 https://adamchanadam.github.io/agent-handoff-kit,依其指引安裝。完成後於你計劃之工作目錄中執行:

npx @adamchanadam/agent-handoff-kit init

執行完畢後,該工作目錄會多出一系列 governance 檔案(AGENTS.md、dev/SESSION_HANDOFF.md、dev/RULE_PACKS.md、dev/PROJECT_INDEX.md、dev/DOC_SYNC_REGISTRY.md 等)。此為 APS 後續步驟所依賴之基礎結構。

第二件 — 安裝 Google Drive 桌面版

  • 前往 https://www.google.com/drive/download/ 下載桌面版
  • 登入後將共享之資料夾右鍵設為「離線存取」(此設定保證 AI 程式無需經網絡即可讀寫 Hub)

第三件 — 安裝 Claude Code

  • 前往 https://claude.com/code 下載
  • 登入後,於你之工作目錄中能開啟新對話;此對話中之 AI 即為 Agent Adam(若你扮 User A)或 Agent Jay(若你扮 User B)

第四件 — 取得本工具(clone repo)

本教學引用之 demo Bridge Pack fixture 位於本 repo 之 examples/ 資料夾。取得方式有兩種:

  • 推薦做法:git clone — 於命令行執行 git clone https://github.com/Adamchanadam/ai-public-squares.git。clone 完成後可於 examples/demo-agent-a/dev/rules/aps-bridge.md 找到 fixture
  • 替代做法:直接下載 zip — 於 GitHub repo 頁面點「Code」 → 「Download ZIP」。解壓後 fixture 同樣位於 examples/ 中
4 件事尚未完成?各個官方網頁皆有逐步指引。先完成全部 4 件,再返回繼續第 3 節。

3.概念熱身 — 教學中常見之詞

後續各節會出現以下詞;此節一次過解釋,後面遇到時不再重複定義。

詞意思
Hub(共享資料夾) Google Drive 上雙方都見到之資料夾,固定名為 AI_Public_Squares。雙方 AI 程式皆視為本地資料夾直接讀寫,Google Drive 處理同步。
Bridge Pack(橋接小規則) 一份 markdown 規則檔(dev/rules/aps-bridge.md),教 AI 程式如何解讀 Hub 上之檔案。每位用戶之工作目錄都需要一份。此教學第 5 / 6 節即係安裝此檔。
packet(交接包) 一個資料夾,內含一份 packet.md(交接之主內容)+ 可選 attachments/。一旦發出即不再修改;若需更新則出 v2 並標明取代哪一版。
outbox(出包紀錄) 每位用戶在自己 lane 之 outbox.log.md,逐行紀錄每次出包、改版、收結、撤回。append-only。
ack(消化紀錄) 每位用戶之 _ack/<agent_id>.ack.json,記下自己消化過對方哪些 packet。
lane(通道) Hub 上之子資料夾 from_<agent_id>/。每位用戶各自寫自己 lane,不寫對方 lane。Drive 同步衝突由此結構性避免。
Agent Adam / Agent Jay 運行於 Adam / Jay 部機之 AI 程式(本教學中用 Claude Code)。與 Adam / Jay 用戶本人之動作分開記錄。
doctor(健康檢查) Agent Handoff Kit 之健康檢查命令(npx @adamchanadam/agent-handoff-kit doctor),驗證 governance 檔案結構正確。每次設置完應跑一次確認綠燈。
kit Agent Handoff Kit 之簡稱。

4.先傾妥幾件事

不動任何檔案。十五分鐘左右之電話或 WhatsApp 傾妥即可 — 你哋實際要拍板者,只有三件;其餘三件皆有合理默認,沿用即可。此步為用戶之間之事,不涉及兩個 AI 程式。

須拍板之三件事

須傾之事建議備註
項目代號 選新之 snake_case 短名(例:branding_2026、logo_refresh) 選全新 slug 最為乾淨 — 不需清任何已有 lane 資料,協定亦不受影響。若想沿用某個共享資料夾中已有之 slug(例如曾做過試演或過往項目使用過),須先清舊資料(刪舊 packets、reset outbox)— 多了一個 step,通常不划算。樣本之完整形狀規則見第 1 節「Placeholder 樣本」。
Jay 部機路徑 Jay 自選工作目錄並於自己部機 mount Hub 例:工作目錄 D:\Work\[項目代號]_Jay\(須完成 kit 之初始化,doctor 通過);Hub 之磁碟字母可能不同(例 H:\ 而非 G:\),此為合理 — 小規則檔內各自寫實際路徑,協定本身不受影響。
大檔試跑是否進行 建議進行,可推遲 第一份真實之逾 50 MB 資產出現之前必須進行一次。若近期皆為文字材料,可遲一步補做。

預設默認之三件事(不需改即可)

項目默認值何時須改默認
Adam agent 名 自選(snake_case,1-30 字內;MVP fixture 使用 adam 做範例) 選短、stable、容易讀即可;無需跟 fixture。
Jay agent 名 自選(snake_case;MVP fixture 使用 jay 做範例) 同上。
共享資料夾中已有之 lane 資料 留底不動(選新 slug 之前提下,已有 lane 自然不受影響) 只有選沿用某個已有 slug 之時才有得選清 / 留;選新 slug 即不關事。若共享資料夾剛啟動、未有任何 lane,此項 N/A。
Adam 部機 Hub 路徑與 Drive 分享權限 Adam 那邊部機掛載 Hub 之實際絕對路徑([Adam 部機之共享資料夾絕對路徑])+ Drive 上層 AI_Public_Squares 資料夾 share 予 Jay 有寫權 若 Drive sharing 設於 slug 子資料夾而非 AI_Public_Squares root,新 slug 須新增 share。若 Hub 尚未掛載或 Drive 尚未 share,須先補完此兩件機器 setup 方能繼續。
雙方傾妥之後

Adam 打開技術版計劃文檔最末之參數表,將拍板之事與確認之默認填齊。此刻即此節完成。

5.Adam 接駁自己部機

此節 Adam 於自己部機,要求 Agent Adam 協助執行;Jay 無需在場。完成後 Adam 之工作目錄已接好橋接小規則,但尚未交收任何 packet。

有 helper script(tools/aps-onboard.ps1)可一次過落 4 個檔案編輯(步驟 2-5 之機械部分 — Identity 連同 4 個 governance row 一齊裝)。詳見技術版計劃之 Tooling shortcut 段。步驟 1(read-only baseline check)與步驟 6(SESSION_HANDOFF Durable Anchors,涉判斷)仍須手做。下面手做版本仍為 authoritative reference,執行 script 之前可先閱讀明白每步在做什麼。
Adam與Agent Adam逐步動作

  1. Agent Adam 做唯讀檢查

    此步之目的:正式驗證 §2 之 4 件先決事完整、kit 健康、Claude Code 可與 Agent Adam 對話。Agent Adam 不會改任何檔。

    1.1 打開命令行

    Windows:打開 PowerShell 或 Windows Terminal。macOS / Linux:打開 Terminal。

    1.2 cd 入你之工作目錄

    於命令行執行(將路徑換為你 §3 中創建之實際路徑;Windows 範例):

    cd C:\Work\branding_2026

    macOS / Linux 範例:cd ~/Work/branding_2026。如尚未創建工作目錄,先返回 §3「準備你嘅項目工作目錄」(待加入,§5 完成後 batch 補)完成創建。

    1.3 啟動 Claude Code 對話

    於工作目錄中執行:

    claude

    此命令啟動 Claude Code 對話 session。後續所有與 Agent Adam 之互動皆於此對話中進行。如命令行報「未找到 claude 命令」,§2 第三件之 Claude Code 安裝未完成,先返回 §2 安裝。

    1.4 將以下 prompt 貼予 Agent Adam

    於 Claude Code 對話中複製貼上以下 prompt(你不需要明白其中之技術細節,Agent Adam 會處理):

    請做以下事但不要改任何檔:

(a) 列出工作目錄之 root 內是否存在以下 5 個檔案:
    AGENTS.md、dev/SESSION_HANDOFF.md、dev/RULE_PACKS.md、
    dev/PROJECT_INDEX.md、dev/DOC_SYNC_REGISTRY.md

(b) 執行 `npx @adamchanadam/agent-handoff-kit doctor`,
    將 output 完整報予我

(c) 執行 `git status`,將 output 報予我;若 git 未初始化,
    告知我並建議跑 `git init`

報告 (a) (b) (c) 之結果,然後停手等我指示。

    1.5 Agent Adam 報出之內容應對照以下表

    檢查項預期結果若不符合,如何處理
    (a) 5 個 governance 檔案 5 個 file 皆存在 §2 第一件之 init 命令未跑齊。返回 §2 第一件補做。
    (b) doctor output Status: passed,通過 34 / 34 項 見下方「若 doctor 失敗」
    (c) git status 顯示分支與工作樹狀態(已 init);或 Agent Adam 建議 git init(未 init) 未 init 可選,但建議跑 git init,APS 之檔案改動方可追溯
    若 doctor 失敗 / 通過項少於 34:常見原因為 kit 版本舊於 0.1.7。Doctor output 首行顯示 kit version。若版本顯示為 0.1.5 或更舊,將以下 prompt 貼予 Agent Adam: 
    請執行 npx @adamchanadam/agent-handoff-kit@latest upgrade,將完整 output 報予我。
    Upgrade 完成後重做 1.4 步驟,doctor 須通過 34 / 34 方可繼續第 2 步。其他失敗原因,將 doctor output 連同 Agent Adam 之報告貼予 Agent Handoff Kit 嘅 GitHub issue 求助。
    5 個 governance 檔案皆存在;doctor 通過 34 / 34;git 已 init(或已決定不 init);Agent Adam 無做任何檔案改動。完成此步即可進入下一步(複製 Bridge Pack)。

  2. 要求 Agent Adam 抄此 repo 隨附之 demo Bridge Pack 副本至工作目錄

    Adam 於對話中要求 Agent Adam 將本 repo examples/demo-agent-a/dev/rules/aps-bridge.md 抄至工作目錄之 [工作目錄]\dev\rules\aps-bridge.md(此即 Adam 角色之 Bridge Pack fixture,procedural 內容與 Jay 角色完全相同,只是 Identity 段不同)。除最上之 Identity 段之外,內容不變。

    Identity 段改為 Adam 部機之實際值:

    - agent_id: adam
- project_slug: [項目代號]
- hub_root: [Adam 部機之共享資料夾絕對路徑]
- other_agent_id: jay

    複製出來後,將 [項目代號] 換為第 4 節拍板之值(例 branding_2026);將 [Adam 部機之共享資料夾絕對路徑] 換為你部機掛載 Hub 之實際 absolute path(例 G:\Cloud\AI_Public_Squares),然後寫入規則檔。完整形狀規則見第 1 節「Placeholder 樣本」。

    將工作目錄之小規則檔與 examples/demo-agent-a/dev/rules/aps-bridge.md 比較,差異只應於 Identity 段。

  3. 要求 Agent Adam 在路由表中註冊此小規則

    於 dev\RULE_PACKS.md 之路由表加一行(## Routing Rule 之前),要求 Agent Adam 抄試演該行原文。

  4. 要求 Agent Adam 將共享資料夾登記為 External Source

    於 dev\PROJECT_INDEX.md 之 External Sources 表加一行,將共享資料夾路徑與寫權範圍寫下。

  5. 要求 Agent Adam 在同步登記表加兩行

    於 dev\DOC_SYNC_REGISTRY.md 加兩行(「出包」一行 + 「消化登記」一行),抄試演原文即可。

  6. 要求 Agent Adam 更新交接檔之 Durable Anchors

    於 dev\SESSION_HANDOFF.md 之 Durable Anchors 與 Active Objective 各補一句,讓將來開啟新對話之 Agent Adam 一讀即知工作目錄已接通共享資料夾,啟動與收尾須跑兩步。

    於 dev\SESSION_HANDOFF.md 中搜「APS」至少有兩處提及。

所有改動以小 commit 形式記錄,標題以 feat(aps):(新檔)或 docs(aps):(governance 改動)為 prefix。Agent Adam 撰寫 commit message,Adam 確認後 commit。

此節完成之後:Adam 之工作目錄已接通,但因 Jay 那邊尚未接通,任何試出包皆未有 receiver。先等 Jay 完成第 6 節,然後第 7 節雙方一同跑。

6.Jay 接駁自己部機

Jay 那邊亦可使用同一個 helper script(tools/aps-onboard.ps1)(Adam 整完之後以 WhatsApp / Drive 連同 demo Jay pack 一齊傳給 Jay 執行)。詳見技術版計劃之 Tooling shortcut 段;Jay-side 之 invocation 以 -AgentId "<your_agent_id>" -OtherAgentId "<counterpart_agent_id>"(以本教學之 narrative 計,分別為 jay 與 adam),demo Jay pack 作為 -DemoPackPath。下面手做版本仍為 authoritative。

此節 Jay 於自己部機,要求 Agent Jay 協助執行。Adam 之角色為預先以 WhatsApp 將試演之 Jay 版本之小規則檔與三行參數傳給 Jay,隨後隨時答問。Adam 與 Agent Adam 皆不可代 Jay 跑命令。

Adam預先準備

將兩件事傳給 Jay:

  • 本 repo 隨附之 demo Bridge Pack(Jay 角色之 fixture),路徑為 examples/demo-agent-b/dev/rules/aps-bridge.md — 與 Adam 角色版本之 procedural 內容相同,只是 Identity 不同
  • 一張短便箋,記錄第 4 節拍板之七行參數結果
Jay與Agent Jay逐步動作

  1. Jay 自行跑部機前置 checklist

    逐項打剔:Google Drive 桌面版已 sign-in、共享資料夾可見、共享資料夾已設「離線存取」、工作目錄已選好並完成 kit 初始化、doctor 通過。

    完成後以一句 WhatsApp 答覆 Adam,例:checklist OK · 工作目錄: D:\Work\[項目代號]_Jay(將 placeholder 換為你部機之實際路徑)。

  2. Jay 要求 Agent Jay 接駁小規則檔

    將 Adam 傳來之 fixture 儲存為 [工作目錄]\dev\rules\aps-bridge.md,要求 Agent Jay 改 Identity 段:

    - agent_id: jay
- project_slug: [項目代號]
- hub_root: [Jay 部機之共享資料夾絕對路徑]
- other_agent_id: adam

    複製出來後,將 [項目代號] 換為第 4 節拍板之值(例 branding_2026);將 [Jay 部機之共享資料夾絕對路徑] 換為你部機之實際路徑(例 D:\Work\AI_Public_Squares;磁碟字母可能為 H:\、D:\ 或其他)。完整形狀規則見第 1 節「Placeholder 樣本」。

    隨後要求 Agent Jay 做與第 5 節步驟 3-6 對應之動作:路由表加一行、External Source 加一行(寫權範圍寫「永不寫 from_adam 與 _ack\adam.ack.json」)、同步登記表加兩行、交接檔更新 Durable Anchors。

    Jay 跑 doctor 仍全部通過。以一句 WhatsApp 答覆 Adam:接駁好,doctor OK。
此節完成之後:雙方工作目錄皆已接通,但共享資料夾上尚未有真實 packet 過。第 7 節即為首次跨機交接之執行。

7.首次跨機交接

由 Adam 發第一個真實 packet 開始,經 Agent Adam 寫入 Hub,Adam 以一句 WhatsApp 通知 Jay,Jay 要求 Agent Jay 撿包,Jay 完成後要求 Agent Jay 出回包,Jay 以一句 WhatsApp 通知 Adam,Adam 要求 Agent Adam 撿到回包並收結。選一個小型內容(一段品牌語氣 note、一個 logo spec 段落)做第一次,不要試大檔(該檔之測試為第 8 節)。

Adam 用戶 Agent Adam 程式 Agent Jay 程式 Jay 用戶
敏感資料禁區。Packet 中不放 credentials、API key、未公開財務、未授權人名。共享資料夾屬中立區,不適合留 secret。如需交付敏感資料,行另一條 secure channel(WhatsApp 私訊、1Password share 等),packet body 中以抽象 reference 指過去,不貼實際值。
用戶決定,Agent 執行 — 之間有一格。Agent Adam 與 Agent Jay 撿到 packet 之後,皆為向對應用戶 propose 待辦,不自動修改本地 SSOT 之檔。用戶決定做與不做、決定何時做、決定做多少,然後再要求 agent 動手執行。Packet body 說「請更新 X」並不等於對方 agent 自動更新 X — 「propose → 用戶拍板 → execute」為刻意保留之人手 step,讓用戶控住每次工作流。
首次跨機驗證 vs 日常運作。下面之完整十步為跨機接駁計劃第一次跑時所做之動作 — 包括 UTC 時間記錄(量度 Drive 同步延遲)與 WhatsApp 完整模板。此皆屬一次性驗收,跑通之後即有 baseline。日常運作精簡甚多 — WhatsApp 短訊一句、receiver 對著現有對話講 check Hub 即可,詳見 §9。

四條 lane,實邊框為人,虛邊框為程式。時間由上至下,十步走完一輪。藍色屬 Adam 域,橙色屬 Jay 域。

Adam與Agent Adam· 發第一個真實 packet(步驟 ①–③)
  1. Adam 於工作目錄開啟新對話,寫好想交予 Jay 之內容。
  2. Adam 對話收尾時要求 Agent Adam 依收尾流程出包:Agent Adam 於共享資料夾下 [項目代號]\from_adam\packets\[時間戳]__[題目]__v1\ 寫 packet.md;append 一行入 from_adam\outbox.log.md;本地 SESSION_LOG.md 加同步行。
  3. (首跑驗收一次性) Adam 自行記下出包之時刻之 UTC 時間,稍後計算 Drive 同步延遲作為接駁計劃之驗收。日常 round-trip 無需做此。
  4. Adam 親自以 WhatsApp 發予 Jay,貼以下短訊([項目代號] 換為實際值)。WhatsApp 不由 Agent Adam 發 — 由 Adam 親手發:
APS Hub has new traffic. Open a new session and paste this to your agent:
"Read <hub_root>/[項目代號], follow APS PROTOCOL, process unconsumed items in from_adam."

複製出來後,將 [項目代號] 換為第 4 節拍板之實際值(例 branding_2026),然後發予 Jay。<hub_root> 無需換 — 此字 literal 保留,Jay 之 Agent Jay 會以自己部機之實際 hub_root 解讀。

Jay與Agent Jay· 撿包、執行、出回包(步驟 ④–⑧)
  1. Jay 收到 Adam 之 WhatsApp,等待 Drive 同步少許時間(通常秒至一分鐘,慢者五分鐘以內皆正常)。
  2. Jay 於工作目錄開啟新對話,將 Adam 之短訊模板貼予 Agent Jay。
  3. Agent Jay 依啟動流程:讀 from_adam\outbox.log.md 與自己之 _ack\jay.ack.json,差集計算 → 列出一個待辦項:Adam 之 packet,連同包中所寫之內容範圍與須執行之項目。
  4. 留意:Jay 於此新對話中並未解釋任何上下文,Agent Jay 自行讀畢。
  5. Jay 執行工作。完成後要求 Agent Jay 依收尾流程出回包:寫 from_jay\packets\[時間戳]__[題目]_reply__v1\packet.md;append from_jay\outbox.log.md;update _ack\jay.ack.json(加一行記下消化了 Adam 之 packet)。
  6. (首跑驗收一次性) Jay 記下 Agent Jay 偵測時刻與出回包時刻之 UTC 時間,稍後計算 Drive 同步延遲。日常 round-trip 無需做此。
  7. Agent Jay 於啟動與 mid-session check Hub 時皆已自動掃描共享資料夾;若無衝突檔報告即可繼續。若有報告,停手依 §11.1 處理(用戶無需自行跑 find)。
  8. Jay 親自以反方向之 WhatsApp 發予 Adam:
APS Hub has new traffic. Open a new session and paste this to your agent:
"Read <hub_root>/[項目代號], follow APS PROTOCOL, process unconsumed items in from_jay."

同上 — [項目代號] 換為第 4 節拍板之實際值(例 branding_2026);<hub_root> literal 保留。

偵測順利、Drive 延遲少於五分鐘、共享資料夾無衝突檔。Jay 之 SESSION_LOG.md 加同步行。
Adam與Agent Adam· 收結整個交接(步驟 ⑨–⑩)
  1. Adam 等 Drive 同步,於工作目錄開啟新對話,將 Jay 之短訊模板貼予 Agent Adam。
  2. Agent Adam 依啟動流程撿到 Jay 之回包,讀畢內容範圍與回包內容,呈現予 Adam。
  3. Adam 要求 Agent Adam 做兩件事:(a) update _ack\adam.ack.json 加一行記下消化了 Jay 之回包;(b) append 一行 close 事件入 from_adam\outbox.log.md,收結 Adam 當初發之 packet:
[時間] | close | [時間戳]__[題目] v1 | reason:jay ack'd and replied
  1. Adam 再開啟一次新對話試。Agent Adam 啟動流程應報「無待辦」 —— 因 close 已將整個 thread 由計算中除去。Agent Jay 那邊亦同樣無待辦狀態。
雙方消化記錄表各有一行;雙方事件記錄皆有對應出包行(Adam 那邊多一行 close);共享資料夾無衝突檔;之後再開啟新對話兩個 agent 皆為「無待辦」。整個交接完成。
頭次之總時間:Adam 由出包時刻至收結時刻之 wall-clock,試演同機為即時,真實跨機會包括兩次 Drive 同步 + Jay 之工作時間,可能數小時至數日不等。Jay 何時開啟新對話予 Agent Jay,由 Jay 用戶決定,並非 Agent Jay 主動 poll。

8.大檔試跑(可選)

驗證「逾 50 MB 之檔不放入包內,改放外面再於包中說明位置」此條規矩。試演未試過,第一份真實大檔出現之前必須試一次。

Adam與Agent Adam動作
  1. Adam 自行準備一份逾 50 MB 之 dummy 檔。可為 padded zip、真實大 PSD,或 PowerShell 執行 fsutil file createnew test.bin 62914560 製造一個 60 MB 之空檔。
  2. Adam 自行將該檔放入共享資料夾之 sibling 位:[共享資料夾]\_external\[題目]\[檔名](若 _external\ 不存在則建立)。
  3. Adam 要求 Agent Adam 出一個 packet,packet 中之 YAML ssot_refs: 寫回此絕對路徑:
ssot_refs:
  - "[Hub 絕對路徑]\\_external\\[題目]\\big_asset.psd"

複製出來後,將 [Hub 絕對路徑] 換為你部機掛載 Hub 之實際 absolute path(例 G:\Cloud\AI_Public_Squares),[題目] 換為此 packet 之 snake_case 短主題(例 logo_convergence);\\ 雙反斜線為 YAML escape,不要改為 single backslash。

  1. Packet 內正文寫清楚:檔為何、為何放外面、Jay 應如何打開。Adam 撰寫,Agent Adam 入檔。
  2. Agent Adam 收尾報告中標明:「Externalized: big_asset.psd (XX MB) at [Hub 絕對路徑]\_external\…」。
  3. Adam 自行以 WhatsApp 通知 Jay,與正常 packet 同。
Jay與Agent Jay收方
  1. Jay 正常開啟新對話、貼模板。Agent Jay 啟動流程列出 packet,同時要報出外置檔之路徑予 Jay 閱讀。
  2. Jay 親手於自己部機 Drive 中以 Explorer / 設計工具打開外置檔,確認檔可讀。Agent Jay 不負責此步,因其只讀寫 Hub 之內檔案。
Packet 內 attachments\ 無任何檔逾 50 MB;外置檔之通知出現於 Agent Adam 收尾報告與 Agent Jay 啟動報告;Jay 可以 Drive mount 打開外置檔。

9.日常使用

第 5 至 7 節跑通,APS 已上線。日常用法為一個重複拍子,每次往返皆為此十步。每個 pill 標明為用戶動作或 agent 動作。

十步走完即一次交接收結。實邊框為人,虛邊框為程式。一日不一定走齊十步,可中途 pause(例:Jay 慢慢做)。

多條交接同時進行則為兩條此十步並行,共享資料夾之事件記錄與消化記錄表會記齊。

日常 WhatsApp 短訊版。日常運作中,WhatsApp 之短訊無需再貼完整模板予對方再貼予 agent。一句「Hub 有新嘢」已足夠通知對方;對方於自己工作目錄之現有對話中講一句「check Hub」即可,Agent 立即撿包(此為默認之日常 primary pattern)。開啟新對話並貼整段模板為 fallback,僅限於第一次跨機(§7)或想重新開始。完整英文模板(§7 之 APS Hub has new traffic...)保留予首次跨機交接做啟蒙教學;日常跑通之後一句「check Hub」即可。
頻率估計:Adam → Jay 方向通常一日一兩次。Jay 回包可能即日內,可能等次日。一條交接之 wall-clock 約 12-48 小時。緊急之交接,WhatsApp 加一句「即刻」即可,協定本身不阻你即時催。

10.中途想交換,不等對話結束

第 7 與 9 節中所講「對話收尾時要求 agent 出包」,此為順手做之時機,並非硬性。Adam 或 Jay 之對話中途任何時刻,只要內容 ready,皆可即時交換。

Adam與Agent Adam· 想中途即時出包

Adam 於對話中途講「我想現在 send 一份予 Jay」,要求 Agent Adam 依出包流程做。Agent Adam 立即寫 packet 入共享資料夾 + 加事件記錄。Packet 為不可變之 snapshot,Adam 之後於對話中之工作,不會倒灌入已出之 packet。

Adam 親自立即發 WhatsApp 模板予 Jay,然後繼續下面之工作。

Jay與Agent Jay· 想中途撿剛到之 packet

Jay 於對話中途收到 Adam 之 WhatsApp,無需 close 對話再開啟新對話。Jay 對 Agent Jay 講一句官方觸發詞:

  • check Hub — 官方觸發詞,最短最直接。Agent 認 case-insensitive(check hub / Check Hub 皆可)。

另兩句同義講法 — 睇下 Hub 有冇新嘢、未消化 — Bridge Pack 皆認,作 fuzzy 兜底(句中含 Hub 連同 新 / 未消化 / check 任一字眼,即觸發同一個 handler)。慣用哪句皆可,但官方一句為 check Hub。

觸發之後 Agent Jay 立即去共享資料夾撿,計算差集,將新待辦呈現予 Jay。Jay 決定做與不做、何時做。

此為手動觸發,並非自動推送。Agent Jay 不主動去 poll 共享資料夾 — 它只於 Jay 開啟新對話啟動時、或 Jay 中途要求一句時方去 check。換言之,Jay 不開對話 / 不要求,Agent Jay 永遠不知有新內容。
真的想自動推送(部機自動彈通知)?此事需要本地 daemon 監察共享資料夾中 outbox 之修改時間,超出純檔案範圍之設計,加重 coupling dependency。設計取捨上,WhatsApp 之人手叫一聲為刻意保留,讓收方控住何時被打斷;自動推送之 cost 為噪音與中斷工作流。值不值得視乎雙方實際使用之頻率與緊急程度,Phase 4 落地之後再評估。

11.出錯處理

11.1 出現 Drive 衝突副本(file (1).md 之類)

理論上不應發生,因有單寫道規矩。Bridge Pack 於每次啟動與 check Hub 觸發時皆自動掃描共享資料夾,掃到任何含 conflict 字眼之檔即時報予你閱讀 — 用戶無需主動跑命令查。若 Agent 報過,或你自行肉眼看到:

  • Adam 或 Jay 不要即時手動刪。
  • 停手,記下衝突檔之完整路徑與名稱。
  • 對照單寫道規矩,找出哪個 agent 程式寫了對方通道之檔(通常此即為根源 — Agent Adam 寫了 from_jay,或 Agent Jay 寫了 from_adam)。
  • 修正根源之後,先決定保留哪個版本(通常為原寫者之版本),衝突副本移走或更名為 .investigated。

11.2 出包但內容錯 / 想撤回

用戶發現錯,用戶與 agent 一同 check 對方是否登記,然後用戶決定,agent 執行。

撤回之行寫法(由 agent 寫):

[時間] | withdraw | [時間戳]__[題目] v1 | reason:作者要求撤回原版

11.3 Agent 撈錯通道(寫了對方通道、寫了對方消化記錄表)

邊界硬規矩應該攔到。若真的漏了:

  • 用戶立即停手,不要繼續要求 agent 出包。
  • 用戶檢查該誤寫究竟為 placeholder(可清)還是真實內容(由用戶決定如何處理)。
  • 用戶決定是否手動清除誤寫;事件 + 原因記入 SESSION_LOG.md(由 agent 寫),讓下次開啟新對話可追溯。

11.4 忘記收結,thread 永遠掛緊待辦

無大事。下次任何一邊開啟新對話,agent 啟動流程列其為待辦,用戶要求 agent append 一行 close 事件即可。close 完之後再開啟新對話即為無待辦狀態。

11.5 Drive 同步延遲逾五分鐘

幾乎一定不是協定問題。Adam 或 Jay 用戶依序檢查:

  1. Drive 桌面版是否正在運行、是否已 sign-in?
  2. 共享資料夾是否已設「離線存取」?若僅為「串流」可能有延遲。
  3. 網絡頻寬是否足夠?大檔則可能須再等。
  4. 對方部機是否 online?Drive 不會主動 push,須 receiver 那邊之 Drive 客戶端 sync 方能見到。
