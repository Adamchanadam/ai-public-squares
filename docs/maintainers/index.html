維護者規格

這頁給開發者與維護者，不給第一次安裝的新用戶。

公眾入口只講如何使用；本頁集中列出內部真源、檢查責任、公開承諾邊界與發佈前應核對的規格位置。

公開承諾邊界

任何公開頁、README、CLI help 或技能提示都不可讓讀者誤以為尚未列入現有能力的事已可使用。

  • 可說：npx aps init 可建立初始設定，CLI 已支援最小收發、修訂、撤回、收結與健康檢查。
  • 必須同時說：每個真實跨機項目仍需各自驗證 Google Drive 同步；外部通知仍由人貼出。
  • 必須標清：0.2.x 主路徑是可靠二人協作。未列入本頁現有能力的事，不可寫成已支援。
  • 不可要求：核心設置不得要求額外雲端開發者專案、外部服務憑證或密鑰設定。
  • 不可說：完全自動、正式生產可用、免檢查、免人工確認、或工具會自動向對方推送通知。
Handoff Kit 整合邊界：APS 不改寫 AGENTS.md managed core。aps init 應只追加可移除的 APS managed block 到 dev\RULE_PACKS.md 與 dev\PROJECT_INDEX.md，並寫入 dev\rules\aps-bridge.md 與 .aps\config.json。日後 uninstall 應移除這些 APS managed block 與 APS 自有檔案，而不碰用戶原有 Handoff Kit 內容。

發佈前最低核對

發佈前不要只看文案是否順眼，要核對工具是否支撐了文案承諾。

  • 公開頁是否只承諾現有可用能力，未把目標體驗寫成已支援。
  • 新手教學是否不再要求閱讀或修改內部規格。
  • CLI 是否能用真實或 throw-away Hub 跑通 init、doctor、publish、inbox、consume、close。
  • aps init 是否已註冊 Handoff Kit 可發現路徑：dev\RULE_PACKS.md 有 APS route，dev\PROJECT_INDEX.md 有 APS installed skill block，且 doctor 會檢查這兩項。
  • HTML 不可連結任何本地 .md 規格檔；需要提及時只用 plain path。
  • npm 版本、GitHub release、README、GitHub Pages 與本地 package.json 邊界一致。

最新 QC 覆蓋範圍

QC 規則以 dev\qc\triggers.md 為單一真源。本頁只提示維護者：功能、流程或教學改動後，以下項目必須納入外發前檢與全面檢。

  • 新手頁的 SVG 宏觀流程圖與能力清單，須與 README、入口頁、CLI、skill spec、QC 真源一致。
  • 日常主路徑須是自然語言請 AI 操作；命令列只可作備用、排錯或維護者驗證。
  • 一語交接、發送前完整性預檢、收件後完整性預檢、補交需求、共識確認、回覆與收結，須在公開文檔、skill 與 QC 真源之間對齊。
  • APS 不自動發 WhatsApp / Email，也不保證對方電腦即時同步；公開文檔與通知文字不得暗示相反能力。
  • 公開 HTML 只寫現有可用能力；未列入現有能力的產品路線名稱留在 Markdown 真源與審核報告，不放入公眾頁面。
  • 若 HTML 參考卡與 dev\qc\triggers.md 不一致，先以該真源為準，再立即同步 HTML。
