Agent Public Square

兩個 AI agent 喺唔同電腦、唔同系統,可以喺同一個項目入面分工協作。

MVP 完成 三層精簡 + QC vocabulary Phase 4 待跑 建立 2026-05-20 最新更新 2026-05-21

一句話講晒

兩位用戶各自用自己嘅 AI agent 做唔同部份(例如一邊寫品牌文字,另一邊做視覺設計),需要交接嘅時候,人只係 WhatsApp send 一句固定模板,搬檔同解釋上下文嘅手工活由 APS 系統嘅結構自己孭。

呢個 repo 係通用 APS template。下面教學用「Adam」「Jay」做兩個 narrative protagonist 講故,佢哋係 example user name — MVP 驗證嗰陣用嗰兩個名,唔係話你一定要用呢兩個名。實際落地時,「Adam」對應你揀做 User A(發起接駁嗰位),「Jay」對應 User B(對方);agent_id、項目代號、工作目錄路徑全部你哋自己決定。

原本嘅痛

未有 APS 前,每次跨機交接都要做晒以下步驟。每個方向都要做一次,即係雙向交流就做兩次。

📦
搬檔
Adam 親手將交接包複製到 Google Drive 嘅指定資料夾。
📱
通知 + 解釋
WhatsApp 通知 Jay,並且解釋呢個 packet 係咩嚟、要做咩。
🤖
Jay 再轉述
Jay 開新 session,要再次將上下文人手 brief 畀自己個 agent。
🔁
每次來回都重做
Jay 出回包又要做返反方向一次:搬檔、通知、轉述。

兩個 agent 之間冇標準 inbox/outbox,Drive 只係檔案倉,唔係協作位;進度、狀態、待處理項目唔同步。

APS 點消除呢啲痛 — 五個招數

1
中立交換區
Drive 上一個共享資料夾(G:\我的雲端硬碟\Adam 工作目錄\AI_Projects\AI_Public_Squares)做中介,雙方 agent 視為本地路徑直接讀寫,完全唔需要 API 或 connector。
2
單寫道
Adam 嘅檔位淨係 Adam 寫,Jay 嘅檔位淨係 Jay 寫,兩個 agent 永不寫對方檔。Drive 同步衝突由結構杜絕,唔靠自律。
3
不可變交接包
每個 packet 一發佈就鎖,要改 = 出新版本(v2、v3)同註明取代邊一版。雙方永遠引到清晰版本,唔會引到對方未定稿嘅中間狀態。
4
焙入啟動讀序
Jay 開新 session 嗰陣,佢個 agent 自己會先去 Hub 撿有冇人留包畀佢,自動列出待辦,Jay 唔使解釋上下文。
5
一句固定 WhatsApp 模板
Adam 出完包,只係 send 一句固定句,Jay 將呢句貼畀自己個 agent。整個過程冇再要任何解釋。

實際跑過一次點樣

2026-05-20 喺 demo agent 跑通,順序如下:

  1. Adam 個 agent 收尾:寫好嘅交接包自動入 Drive 嘅 from_adam/ 道。
  2. Adam WhatsApp Jay:一句固定模板,30 秒事。
  3. Jay 開新 session:個 agent 啟動嗰陣自動撿到 Adam 嘅包,讀晒包入面嘅 scope 同待辦,主動列畀 Jay 睇。
  4. Jay 做完視覺,session 收尾:自動出回包入 from_jay/,並且登記咗已消化 Adam 嗰一版。
  5. Adam 下次開 session:個 agent 自動撿到 Jay 回包,讀完,登記消化,正式收結整個交接 thread。
驗證結果:全程冇人手搬檔、冇人手解釋上下文。Drive 冇出衝突副本。兩邊 agent 嘅待辦清單最後都計算為「空」,系統返到穩定狀態。

Phase 4 等住做嘅嘢

下次想真係用 APS 同 Jay 對接,以下幾項要逐項收口。每一項喺 Phase 4 計劃文檔都有完整 step-by-step 任務拆解 + 驗收條件,可直接照做。

  • T0 鎖定參數(經 Layer 2 簡化後只剩 3 個拍板):你揀新 project slug(建議 branding_2026 一類新名,demo lane 自然唔受影響)+ Jay 報自己部機嘅工作目錄同 Hub 路徑 + T10 大檔試跑做唔做。其餘四項(雙方 agent_id / demo 舊資料處理 / Adam Hub + 共享權限)默認沿用即可。
  • T0b(preferred prerequisite):喺兩個 demo workspace 嘅 session 入面,將 Bridge Pack startup 兩個 polish 落入 demo packs(衝突檔自動掃 + 官方觸發詞 check Hub 認 case-insensitive 連 2 個同義詞兜底),咁 Block 4A / 4B 嘅「copy demo pack verbatim」步自然繼承,真實 runtime 唔需要再 patch。
  • Block 4A(可獨力先做):將 Bridge Pack 接入 User A 自己揀嘅真實項目工作目錄。可選跑 tools/aps-onboard.ps1 一次過落 T2-T5 嘅 4 個機械 file edit(idempotent + DryRun 可預覽 + 失敗有 manual fallback),T6 SESSION_HANDOFF Durable Anchors 仍手做(涉判斷);或者按 Phase 4 計劃逐 task 手做。
  • Block 4B(等 Jay 配合):Jay 部機 mount 同一個 Drive 資料夾、右鍵設「離線可用」、接 Bridge Pack。Adam 可以將 tools/aps-onboard.ps1 經 WhatsApp / Drive 傳俾 Jay,Jay-side 同樣一次過跑(調 -AgentId "jay" -OtherAgentId "adam")。
  • Block 4C:由你 + Jay 真跑一次跨機 round-trip,觀察 Drive 同步延遲、確認無衝突副本。Block 4A 嘅 first run 亦係 tools/aps-onboard.ps1 嘅 first real insertion-path verification。
  • Block 4D(建議,但可推遲):大附件(>50 MB)dry-run,驗證 externalization 規則。
