公開入口

讓兩部電腦上的 AI 有一個共同交接處。

AI Public Squares 用 Google Drive 上的一個共享資料夾，替兩邊的 AI 留下交接包、上下文、版本記錄與收件狀態。它不是伺服器，也不會自動傳訊息；它的價值是讓對方 AI 在檢查 Hub 時讀到完整交接，而不是靠人類重複搬運背景。

npm latest 0.2.7 公眾可用版本

先講清楚：目前可依賴甚麼？

本頁只列公眾可照做的現有流程：安裝、設定、健康檢查、交接、收件、修訂、撤回與收結。

已可實際使用的部分

npx aps init 可安裝 APS 技能、建立 Hub 骨架、寫入本地設定；doctor、publish、inbox、consume、revise、withdraw、close 已可支援最小日常往返。

使用時要知道的邊界

AI 對話中的自然語言引導、錯誤補救、跨機項目首次驗證，仍需要保守試用。工具不會自動推送通知，但 AI 應替你生成可直接複製貼上的 WhatsApp 或 Email 通知。通知應叫對方在自己電腦上使用已接入 APS 的對應項目，不應複製發送方的本機 Google Drive 路徑。

對新用戶的正確期待：今天最可靠的路徑是「照教學安裝 → 由 npx aps init 建立設定 → 打開 AI 說『教我用 APS』→ 讓 AI 協助你做第一次測試交接」。命令列保留作可驗證備用；日常主路徑應是自然語言。
產品邊界：0.2.x 主路徑是二人協作。現在只承諾本頁寫明的安裝、健康檢查、交接、收件、修訂、撤回與收結能力。核心方向是不要求雲端開發者專案、額外 API 權限或密鑰設定。

它解決甚麼問題？

最新版在哪裡？

每次交接都變成一個不可直接覆寫的版本包，之後要修正就出新版本或撤回，不用靠 WhatsApp 記憶。

新 AI 對話忘記上下文？

AI 可以從同一個 Hub 讀到對方留下的交接包、共同目標、各自任務邊界、版本與已讀狀態，減少每次重新交代背景。

Google Drive 衝突副本？

雙方各自只寫自己的通道，避免兩邊同時改同一個檔案。若出現疑似衝突，工具會把它當成需要檢查的狀態。

每次往返重做？

第一次接駁後，你可以對 AI 說「把目前背景交給對方」或「看看對方有沒有回覆」。AI 依本地設定與 Hub 狀態協助你判斷下一步。

第一次使用的最短路徑

你不需要先讀內部規格，也不需要自行編輯任何治理文件。照以下次序走即可。

  1. 在你的項目資料夾打開終端機。
  2. 先安裝 Agent Handoff Kit，讓這個資料夾具備基本交接與健康檢查能力。
  3. 安裝 APS 套件。
  4. 執行 npx aps init，按畫面提問填入五個值。
  5. 執行 npx aps doctor，確認 Hub 與本機設定正常。
  6. 在 AI 工具打開同一個項目資料夾，輸入「教我用 APS」。
npx --yes @adamchanadam/agent-handoff-kit@latest init
npm install --save-dev @adamchanadam/aps@latest
npx aps init
npx aps doctor

前往完整新手教學，按畫面逐步做。

npx aps init 問你的五個值

這五個值不是技術考試，只是讓兩部電腦對準同一個共享空間。

畫面會問意思建議填法
你的角色 A 代表你先建立 Hub 給對方加入；B 代表你收到對方 starter pack 後加入。 第一次建立通常選 A。
項目代號 Hub 內用來分開不同合作項目的資料夾名稱。 aps_uat、branding_2026。只用小寫英文、數字、底線。
你的 agent id 你這邊 AI 在 Hub 內的通道名稱。 通用測試可用 agent_a。
對方 agent id 對方 AI 在 Hub 內的通道名稱。 通用測試可用 agent_b。
Google Drive 本機路徑 你電腦上 Google Drive 同步出來的 AI_Public_Squares 資料夾完整路徑。 請在檔案總管打開該資料夾，複製地址列完整路徑。不要輸入含 ... 的省略路徑。

日常使用應該多簡單？

本工具的目標不是讓你記更多命令，而是讓 AI 可按同一套規矩代你處理交接。命令列保留作可驗證、可追溯的備用路徑；一般使用時應先對 AI 說你想完成甚麼。

你有內容給對方

告訴 AI：「幫我將當前任務整理成 APS 交接包給對方」。AI 應自動讀設定、檢查 Hub、整理共同目標、本方任務、對方任務、交叉點、待處理事項、關鍵檔案與請對方回應的問題。發送前，AI 必須先做完整性預檢並給你確認，然後才發佈交接包並生成可貼上的通知。

對方說有新內容

告訴 AI：「check Hub」。AI 應檢查收件、列出待處理內容，並先判斷交接包是否齊全；若資料不足，AI 應主動整理缺漏並生成補交需求包給對方。

出現錯誤或同步問題

AI 應先跑 doctor，再用官方資料協助你檢查 Google Drive、Connector 或工具安裝問題。

目前限制：APS 不會自動提醒對方電腦有新內容。AI 應替你生成可直接複製貼上的 WhatsApp / Email 通知，你只需貼到平常渠道請對方「check Hub」。通知文字應用項目代號、主題與交接編號識別內容；本機 hubRoot 不應當成對方電腦上的路徑。若 AI 未能順利完成，才回到命令列備用路徑：npx aps doctor、npx aps inbox、npx aps publish、npx aps consume、npx aps revise、npx aps withdraw、npx aps close、npx aps config。

先確認共識，再開始做

若對方交來的內容與你這邊理解不一致，APS 不應讓 AI 直接照做。正確流程是先回饋、先確認，再開工。

AI 應先停下來

它應列出對方說法、本方理解、衝突點、風險與需要對方確認的問題。

回覆一個共識確認包

AI 應用 APS 發出確認包，並生成可直接複製貼上的 WhatsApp / Email 通知。雙方確認共同目標與任務邊界後，才繼續執行。

它不是假設兩邊做同一件事

實際協作中，兩個 AI 可能收到不同 brief，也可能只在一小段流程交會。APS 的交接包應把共同目標與各自邊界寫清楚。

常見情景

一邊整理品牌語氣，另一邊製作網站。交接包要講清楚共同目標是「網站文案符合品牌語氣」，不是要求對方接手整個品牌工作。

流程交叉

一邊做研究，另一邊寫報告。交接包要標明哪些研究結論可用、哪些仍未核實、對方要回覆甚麼。

極端情況

若兩邊 brief 互相矛盾，AI 不應假裝已對齊。交接包應列出差異，要求對方或人類先確認共同目標。

公眾文檔與內部文檔如何分工

公眾頁面只保留新用戶需要知道的事：工具做甚麼、目前能做到甚麼、如何安裝、如何完成第一次測試、遇錯怎樣求助。內部檢查、規格真源與維護者紀錄不放在公眾主導航中，避免新手把維護者流程誤讀成使用流程。

維護者可在 repo 內依內部真源核對規格與檢查責任；公眾入口不要求新用戶閱讀這些材料。
