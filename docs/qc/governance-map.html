分層 QC — 觸發詞彙

三級觸發詞彙將既有 6 層 QC 機制收口成 3 個入口。

本卡片：參考 單一真源：dev/qc/triggers.md 建立：2026-05-21 S6

設計意圖

過往「全套 QC / 完整 QC / 跑驗收 / 整體 audit」此類含混詞容易令人類與 AI 對驗收範圍理解不一致。三級觸發詞彙鎖定三件事:每級的覆蓋範圍與時間、較高級別自動包含較低級別、含混詞觸發停手反問。

本卡片只屬日常閱讀與操作參考;觸發詞、級別定義、覆蓋範圍之單一真源為 dev/qc/triggers.md。若本 HTML 與該檔內容不一致,以該檔為準,並須將本卡片同步修正,不可讓讀者面長期漂移。

三級速覽

級別觸發詞時機時間預算
🟢 快檢 跑快檢 / quick-check / qa-gate session 收工前 / commit 之前 < 2 分鐘
🟡 外發前檢 跑外發前檢 / release-check / handover-verify 推送遠端 / 分享計劃 / 跨工作目錄傳檔之前 5–10 分鐘
🔴 全面檢 跑全面檢 / full-check / full-audit 跨機往返測試 / 協定改版 / 治理工具重大升級之前 30+ 分鐘

三級詳述

🟢 快檢

別名：跑快檢 / quick-check / qa-gate

時機：session 收工前；五區段執行完之後；commit / amend 之前。

覆蓋（4 項）：

  • kit doctor 報 status: passed
  • git status 同 handoff Workspace Identity 一致
  • 當次 session 所有 grep acceptance 全部命中
  • 有 script 改動：parse-check + DryRun against demo

驗收：四項全過。
失敗：修源重跑。

🟡 外發前檢

別名：跑外發前檢 / release-check / handover-verify

時機:物件離開本工作目錄之前(推送遠端 / 分享計劃 / 傳橋接規則包予 Jay)。

覆蓋：快檢 4 項 + 5 項：

  • 跨工作目錄佔位符審核
  • 計劃 / 教學頁 / 橋接規則包三處交叉閱讀
  • HTML preview 視覺確認
  • PII / secrets sweep
  • 語氣與用語紀律掃描

驗收：9 項全過。
失敗：handover stop。

🔴 全面檢

別名：跑全面檢 / full-check / full-audit

時機：跨機往返測試 / 協定升版 / 引入新 AI 程式 / 治理工具重大升級之前。

覆蓋：外發前檢 + 5 項人工 / 跨工作目錄：

  • 跨工作目錄審核
  • MVP 形式往返流程回歸測試
  • 橋接規則包啟動行為記錄
  • 五區段 / 自審 retrospective
  • 審核報告輸出

驗收：14 項 + report committed。
失敗：逐條列阻擋項。

APS 全面檢三條主線

AI Public Squares 的全面檢不只是多跑幾條命令,而是要證明公開說法、外發物可信度與協定實際行為三者一致。狀態須逐項標示為通過、失敗、受阻或接受風險。

主線要核對之事不通過時代表什麼
公開承諾一致性 README、入口頁、教學中心、完整落地教學、本頁、CLI help、skill spec 對「目前可用」「目標體驗」「手動替代路徑」「npm publish 狀態」描述一致。 公開面可能把未發佈或未實作能力寫成已可用。
發佈前可信度 任何會離開本工作目錄的內容,包括 README、GitHub Pages、npm package、Bridge Pack、walkthrough、QC 卡片,皆須讓外部讀者理解並按當前狀態操作。 外人可能照文件操作卻撞上未核實、未發佈或不完整流程。
協定實際運行正確性 Hub 協定、橋接規則包樣板、示範與實際工作目錄、發件記錄、收件記錄、單寫通道、交接包版本、啟動時待處理計算、收結 / 撤回行為須有實際操作記錄或對應工作目錄證據。 文檔說得通,但實際共享資料夾流程未必跑得通。

主要使用流程走通

全面檢須按情景走讀者真正會走的路。不能跑的情景不可當作通過,只能標示為受阻並寫明缺少哪一項證據。

情景驗收焦點主要入口
零認知讀者入口流程 讀者由 README 進入後,能分清「目前可試」與「目標體驗」。 README.md、入口、教學中心
手動設置流程 由前置事項、工作目錄、橋接規則包、治理列到首次交接包,每一步不依賴不存在的 npm package 或私有本機路徑。 完整落地教學
日常協作流程 發出內容、WhatsApp 通知、收方檢查 Hub、收件 / 消化、回覆 / 收結能追蹤同一條邏輯。 完整落地教學第 10 節
出錯補救流程 Drive 同步延遲、衝突副本、寫錯通道、交接包格式錯誤、版本不對齊,皆有偵測、停手、用戶確認、修復或升級路徑。 完整落地教學第 12 節

嵌套規矩

三級嚴格嵌套：

  • 跑外發前檢 必自動跑 快檢 4 項先
  • 跑全面檢 必自動跑 外發前檢(即同時包含快檢)

AI 不可在較高級別檢查被觸發後省略較低級別檢查。

反問規矩

當用戶用以下含混詞，AI 必須停手反問「你指快檢 / 外發前檢 / 全面檢?」：

含混詞對應動作
全套 QC反問
完整 QC反問
做 QC反問
跑驗收反問
Audit / 整體 audit反問
QC(單字)反問

未列於以上 list 之含混表達,按字面 + context 解;如仍有 ambiguity 寧反問。

典型用法

  • 🟢 快檢:commit 前 spot-check;五區段執行完之後之 baseline 驗證;改完 doc 試 anchor 不斷
  • 🟡 外發前檢:push remote 之前;Adam 傳 Bridge Pack 予 Jay 之前;新 Phase plan 分享予外人之前
  • 🔴 全面檢：跨機往返落地之前；協定升版之前；引入新 AI 程式之前；連續多輪工作後重置基線

點呼叫

對住 AI 講一句 trigger keyword 即可：

  • 「跑快檢」— AI 跑 doctor + git status + grep set
  • 「跑外發前檢」— AI 跑快檢 + 五項外發 audit
  • 「跑全面檢」— AI 跑外發前檢 + 五項全面審核;如涉及跨工作目錄,AI 列出需於對應工作目錄另開新對話處理的子任務
講含混詞 = 觸發反問。如果用「做 QC / 跑驗收 / 整體 audit」此類詞,AI 會停手問「你指快檢 / 外發前檢 / 全面檢?」。此乃紀律,不為出 bug — 防止 scope 漂移。
